reference C

omniLua runs at about 1.37× the wall-clock time of reference C across the tracked workloads, or 1.3× with PGO. It's a bytecode interpreter, not a JIT, so it isn't LuaJIT-fast. Memory safety costs almost nothing; the gap to C comes from how values are represented and dispatched.

Where the gap is

Safe Rust is often assumed to be slow because of bounds checks and borrow guards. We measured that by removing them, and it isn't where the time goes.

~0%

Safety tax

Removing bounds checks and RefCell guards recovers almost no wall time, so memory safety isn't the cost. There's no unsafe outside the GC, the dynamic-library loader, and the wasm pointer ABI.

16 vs 8

Representation

Each Lua value is 16 bytes where C packs 8, because NaN-boxing isn't done yet. That wider value is copied on every register move, and it's the biggest part of the roughly 2.3× instruction-count gap on the hot path. NaN-boxing would close most of it.

dispatch

Dispatch & layout

The rest is interpreter dispatch and code layout — the gap between a match loop and C's computed-goto threading. PGO recovers part of it; the remainder tracks the representation work above.

The gap to C is representation and dispatch, and both have known fixes. Neither is the price of safety.

Not a JIT

omniLua is a bytecode interpreter, not a JIT, and it isn't faster than reference C. If you need LuaJIT speed or a mature C binding, use mlua. What omniLua gives you instead is the same runtime natively and on wasm32, in safe Rust, at a competitive speed.
