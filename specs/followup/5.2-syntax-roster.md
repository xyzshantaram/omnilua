# Lua 5.2 — syntax + stdlib roster (the float-only + `_ENV` bridge)

Everything below was reproduced against the unmodified reference binary
`/tmp/lua-refs/bin/lua5.2.4` (`make macosx` build). 5.2 reuses the modern
`_ENV` globals core but is **float-only** (one `number` type, no integer
subtype) and removes `fenv`. It is the bridge from the modern family to the
5.1 legacy family: it needs *no* fenv fork, so it is implemented here by
version-gating the existing modern core, exactly as the 5.1 spike (`d8ec33f`)
sketched for the number model.

`LuaVersion::V52` is now in `is_supported()`; `Lua::new_versioned(V52)` and
`LUA_RS_VERSION=5.2` both run. V51 still refuses (it needs the fenv core).

## Number model: FLOAT-ONLY (the sharp axis)

`LUA_NUMBER = double`, no `lua_Integer` subtype. Observable consequences and
where each is enforced:

| Behavior | lua5.2.4 | seam (this branch) |
|---|---|---|
| `10/2`, `2^2` print as `5`, `4` (no `.0`) | `5` / `4` | `crates/lua-vm/src/object.rs` `number_to_str_buf`: under V51/V52 do not append `.0` to an integer-valued float |
| `tostring(1.0)` == `1` | `1` | same |
| numeric literals are all float | `9007199254740993` → `9.007199254741e+15` | `crates/lua-lex/src/lib.rs` `read_numeral`: under float-only, Int from `str2num` is widened to `Float(i as f64)` |
| `io.write(1.0)` matches `print(1.0)` | `1` | `crates/lua-stdlib/src/io_lib.rs`: `io.write`/`f_write`/`g_write` route numbers through the shared `num_to_string` (`%.14g`) instead of the old always-`%e` path (a *cross-version* bug that only surfaced once every number was a float) |
| `math.type` / `math.tointeger` / `math.maxinteger` / `math.mininteger` / `math.ult` | all `nil` | `crates/lua-stdlib/src/math_lib.rs`: nil these fields under V51/V52 |
| string→number coercion stays float | `"10"+5` → `15` | inherited; observably identical (printed without `.0`) |

**Is gate-based float-only sufficient for 5.2?** Yes, with one production-site
fix that the task anticipated. The dual `LuaValue` enum is kept; internal `Int`
values (string length `#t`, `select('#')`, table positions, `string.format`
`%d`) are harmless because (a) 5.2 has no `math.type` to observe int-ness and
(b) every numeric *render* now goes through the version-aware `num_to_string`.
The one place that bypassed `num_to_string` was `io.write` (used `{:.14e}`); it
is now routed through `num_to_string`, fixing it on **all** versions. No deeper
change (no `LuaValue` fork) is required.

## Syntax: rejected in 5.2 (5.3+ integer operators)

All produce a parse error; messages reproduced byte-for-byte against lua5.2.4
(via `load([[...]])`, so the chunk-name prefix is `[string "..."]`):

| Construct | lua5.2.4 message | seam |
|---|---|---|
| `7//2` (floor div) | `unexpected symbol near '/'` | lexer: under float-only, never form `TK_IDIV` (return bare `/`) |
| `1 << 4`, `256 >> 4` | `unexpected symbol near '<'` / `near '>'` | lexer: never form `TK_SHL`/`TK_SHR` |
| `6 & 3`, `6 \| 3` | `<eof> expected near '&'` / `near '\|'` (or `')' expected` inside a call) | parser: `getbinopr_versioned` returns `NoBinOpr` for `&`/`\|`/binary `~` |
| `5 ~ 3` (binary xor) | `... near '~'` | same |
| `~0` (unary bnot) | `unexpected symbol near '~'` | parser: `getunopr_versioned` returns `NoUnOpr` for `~` |
| `local x <const> = 1` | `unexpected symbol near '<'` | pre-existing: attributes already gated to 5.4/5.5 (`version_has_attributes`) |

## Syntax: present in 5.2

`goto`/labels (`::x::`), `..` concat, the `_ENV` upvalue model. All inherited
from the modern core; verified working.

## Known gaps (honest accounting)

1. **`module` global** — the deprecated 5.2 `module(...)` function is not
   implemented (`type(module)` is `nil`; should be `function`). It manipulates
   the calling chunk's `_ENV` and is essentially dead in modern Lua code. Left
   unimplemented rather than shipping a broken stub.

2. **Error-message word order** — for *runtime* "attempt to call a nil value"
   errors, lua-rs uses the 5.3+ phrasing `attempt to call a nil value (global
   'X')` while 5.2.4 uses `attempt to call global 'X' (a nil value)`. This is a
   **pre-existing, cross-version** difference in the VM call path (it is
   byte-identical on our 5.4 vs lua5.4.7 too — i.e. not a 5.2 regression), and
   is the only reason the `echo`/`env`/`readonly`/`trace-globals` 5.1-test
   scripts diff (their underlying behavior — `getfenv`/`table.getn` being nil —
   is correct for 5.2). Fixing it is a separate error-message-format seam.
