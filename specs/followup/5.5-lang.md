# Lua 5.5 language-feature divergences (DISCOVER, read-only)

Probed `lua5.5.0` (`/tmp/lua-refs/bin/lua5.5.0`) against `lua-rs`
(`LUA_RS_VERSION=5.5 target/debug/lua-rs`) for the four globalstat / vararg /
attribute / for-loop areas. Every repro below was reproduced with
`specs/oracle/diff_one.sh 5.5 '<code>'` or direct binary invocation, and the
5.5-only paths were confirmed to NOT exist on 5.4/5.3 (so a fix must stay gated
to `LuaVersion::V55`).

Note on the probe preamble: in 5.5 strict mode (after the first `global` decl in
a scope) every free name must be declared, so the repros declare the builtins
they use, e.g. `global print; ...`.

## Summary

5 distinct divergences, in 3 categories:

- **Missing runtime guard** (1): `global x = expr` does not raise
  `global '<name>' already defined` when the global is already non-nil.
- **Missing syntax** (2): named varargs `function f(...t)` and the
  `global function f() ... end` declaration form are both rejected by the
  lua-rs parser.
- **Error-message text** (2): `<close>` on a global and `unknown attribute`
  produce the wrong message string (right exit code, wrong text), and the
  message-location formatting is also wrong on the local-attribute path.

For-loop control-variable read-only (item 4) is FULLY CORRECT — no divergence.

## Confirmed-correct (no action)

- **For-loop control var read-only** (numeric and generic): assigning to the
  loop var is the compile error `attempt to assign to const variable 'i'`.
  `for i=1,3 do i=10 end`, `for k,v in pairs({a=1}) do k=2 end`, and the
  closure form all `MATCH` on both 5.5 and 5.4.
- **`<const>` global read + reassign**: `global x <const> = 1; print(x)` → `1`;
  `global x <const> = 1; x = 2` → compile error
  `attempt to assign to const variable 'x'` (MATCH).
- **Strict-mode undeclared free name** → `variable '<name>' not declared`
  compile error (MATCH).
- **Block-scoped `global` decl**: a `global` decl is confined to its block;
  outside the block the name is undeclared (MATCH, K-case below).
- **`_VERSION`** is correct when reachable (MATCH).
- **`global` is a plain identifier on 5.4/5.3** (not a keyword): all 5.5 forms
  are correctly inert pre-5.5 (MATCH on `global = 5`, `...t`, etc.).

## Divergence 1 — `global '<name>' already defined` runtime guard MISSING (clear-cut)

This is a **runtime** error, not compile-time. The exact rule, derived from
`/tmp/lua-refs/lua-5.5.0/src/lparser.c` (`globalnames`/`initglobal`/`checkglobal`,
emitting an `OP_CHECKGLOBAL`) and `ldebug.c:817 luaG_errnnil`:

> When a `global name = expr` declaration (WITH an initializer) executes, it
> first checks the *current runtime value* of the global. If it is **non-nil**,
> raise `global '<name>' already defined`. A `global name` WITHOUT an
> initializer never checks. The check is on the live value, not on
> compile-time redeclaration tracking.

Repros (all expect ref exit 1, traceback ending `[C]: in ?`):

| Code | ref 5.5.0 | lua-rs 5.5 | verdict |
|---|---|---|---|
| `global print; global x = 1; global x = 2; print(x)` | error `global 'x' already defined` @ line of 2nd init | prints `2`, exit 0 | DIFF |
| `global print; global x; x = 5; global x = 9; print(x)` | error (x is 5, non-nil) | prints `9`, exit 0 | DIFF |
| `global print; global x = 1; do global x = 2 end; print(x)` | error | prints `2`, exit 0 | DIFF |
| `global print; global function f() end; global function f() end` | error `global 'f' already defined` | (also blocked by Div.2b) | DIFF |
| `global print; global x = 1; x = nil; global x = 2; print(x)` | prints `2` (x was nil'd) | prints `2` | MATCH — proves it is a live-nil check, not redeclare tracking |
| `global print; global x; global x; print(x)` | prints `nil` (no init → no check) | prints `nil` | MATCH |
| `global print; global x = 1; x = 2; x = 3; print(x)` | prints `3` (plain assigns, no check) | prints `3` | MATCH |
| `global print; global f; function f() global x = 1 end; f(); f(); print(x)` | `variable 'x' not declared` (block scope) | same | MATCH |

The error line is the line of the offending **initializer** (verified
multi-line: error reported at line 3 for the 2nd `global x = 2`).

- **Expected message**: `(command line):<line>: global '<name>' already defined`
  followed by a normal runtime traceback (`stack traceback:` /
  `(command line):<line>: in main chunk` / `[C]: in ?`).
- **Impl location**: `crates/lua-parse/src/lib.rs` `globalstat` (lines ~4090-4176;
  the initializer path is the parse-and-store added by `v55_global_initializer_stored`).
  Needs a pre-store "is current value nil" guard. Upstream uses a dedicated
  `OP_CHECKGLOBAL` opcode (`lua-code`) feeding `luaG_errnnil`; the lua-rs
  equivalent could emit the same guard inline in codegen, or perform the check
  in the store path. Cross-check the GC/runtime error helper for the traceback
  shape.
- **Clear-cut?** Behavior + message are unambiguous and fully pinned. The only
  design choice is opcode-faithful vs inline-check; either is acceptable since
  the oracle only sees behavior + message. **CLEAR-CUT.**

## Divergence 2a — named varargs `function f(...t)` syntax MISSING (clear-cut spec, medium effort)

`...t` is valid 5.5 syntax and binds the trailing varargs into a fresh **packed
table** `t` (exactly `table.pack(...)` semantics: 1-based sequence PLUS a `.n`
field counting all args including nil holes). `...` still works inside the body
alongside the named form.

Confirmed semantics (ref 5.5.0):

| Code | ref result | meaning |
|---|---|---|
| `local function f(...t) return t end; local r=f(1,2,3); print(#r,r[1],r.n)` | `3  1  3` | t = packed sequence |
| `local function f(a,...t) print(a,#t,t[1],t[2]) end; f(10,20,30)` | `10  2  20  30` | extra args after fixed params |
| `local function f(...t) return ... end; print(f(1,2,3))` | `1  2  3` | `...` still usable in body |
| `local function f(...t) print(...) end; f(1,2,3)` | `1  2  3` | both coexist |
| `local function f(...t) print(#t, t.n) end; f(1,nil,3)` | `1  3` | `.n` counts nils; `#` is a border |
| `local function f(...t) print(t.n) end; f()` | `0` | `t.n == 0`, t is an empty table |
| `local function f(...t) return t end; print(f(1)==f(1))` | `false` | fresh table per call |
| `local function f(...t) t[1]=99; return t[1] end; print(f(1,2))` | `99` | t is mutable |
| `local function f(...t <const>) return t end` | `')' expected near '<'` | NO attribute allowed on `...t` |

lua-rs 5.5 rejects ALL of these at parse time: `')' expected near 't'`.

- **Impl location**: parser `parlist` at `crates/lua-parse/src/lib.rs:3121-3154`.
  Currently the `TK_DOTS` arm (line 3132) sets `isvararg=true` and breaks; it
  must optionally consume a following `TK_NAME` and bind it as a vararg-table
  local. Upstream (`/tmp/lua-refs/lua-5.5.0/src/lparser.c:1065 parlist`): after
  `...`, if next is `TK_NAME` it does `new_varkind(ls, name, RDKVAVAR)`, else a
  hidden `(vararg table)` local; then `setvararg` + `adjustlocalvars(ls,1)`.
- **Codegen/VM note (the non-trivial part)**: upstream represents the vararg
  table *virtually* and only materializes a real table when it is indexed or
  mutated (`lparser.c:281-318 check_readonly` / `VVARGIND`/`VVARGVAR` exprkinds,
  `needvatab`). A faithful port needs the new expr kinds + a `needvatab` flag on
  the proto, OR a simpler always-materialize-via-`table.pack` lowering at
  function entry (behaviorally equivalent for the oracle, modulo identity which
  is per-call fresh anyway). The simpler lowering is the pragmatic path.
- **Clear-cut?** The *spec* is clear-cut and fully pinned. The *implementation*
  touches `lua-parse` codegen + `lua-vm` vararg handling, so it is the largest
  item here. **CLEAR-CUT SPEC, MEDIUM-HIGH IMPL EFFORT.**

### LANDED (pragmatic always-materialize lowering) + ONE DEFERRED ALIASING GAP

Implemented the simpler lowering on `finish-5.5`:

- Parser `parlist` (`crates/lua-parse/src/lib.rs`): after `...`, on `V55` only,
  consume a following `TK_NAME`, push it as a local (mirroring upstream's
  `new_varkind(..., RDKVAVAR)` + separate `adjustlocalvars(1)`), single combined
  `reserveregs`. On 5.4/5.3 the name is never consumed, so the syntax stays the
  reference parse error (`')' expected near 't'` / `... near '<'` for an
  attribute) — CI-asserted in `named_varargs_rejected_pre_55`.
- New opcode `OP_VARARGPACK` (= 84; `lua-vm` `OpCode`, `lua-code` `OP_MODES` /
  `OPNAMES`, both round-trip tests bumped to 85). Emitted once at function entry;
  packs the frame's `nextraargs` into a fresh table (`new_table_with_sizes` +
  `raw_set_int` + a `.n` integer field) — i.e. `table.pack(...)` semantics.
- A **shared-core** fix fell out: `cg_set_returns` for the `VARARG` case did
  `freereg += 1` without `bump_maxstack`, leaving the high-water mark stale and
  tripping the `maxstacksize >= freereg` parser invariant whenever the VARARG sat
  above register 0 (now true for a named-vararg local). Upstream
  `luaK_setreturns` uses `luaK_reserveregs(fs,1)`, which bumps maxstack. Fixed to
  match. Verified non-regressing on 5.4/5.3 (`check.sh` green; vararg-splat
  `diff_one.sh` MATCH).
- CI (`multiversion_oracle.rs`): `v55_named_varargs` (#t, `.n` incl. nil holes,
  empty `f()` → `.n==0`, `a,...t` extra-after-fixed, `...` still usable,
  fresh-per-call, `t` mutable) + `named_varargs_rejected_pre_55`. All green; all
  values captured from `lua5.5.0`.

**DEFERRED — vararg-table / `...` aliasing.** In upstream 5.5 the named table and
`...` share ONE storage object: mutating `t` is observable through a later `...`.
Repro (ref 5.5.0 vs lua-rs):

```
local function f(...t) t[1]=99; return ... end print(f(1,2,3))
  ref : 99  2  3      (… reflects the mutation to t)
  ours: 1   2  3      (… is an independent snapshot)
```

This is the only failing case in official `vararg.lua` (line 111: an `aux` that
does `for k,v in pairs(v) do t[k]=v end; return ...`). The pragmatic
always-materialize lowering makes `t` and `...` independent; every
non-mutate-then-splat case matches. A faithful fix needs upstream's virtual
vararg table — the `VVARGVAR`/`VVARGIND` expr kinds + a `needvatab` proto flag so
`...` reads the *same* object the table mutations land in (or, minimally, a proto
field carrying the vararg-table register so the `VARARG` opcode unpacks that
table instead of `nextraargs`, plus bytecode dump/undump for the new field).
That is the architectural piece flagged "MEDIUM-HIGH IMPL EFFORT" above and is
left unimplemented rather than half-done. Re-entry point: add the proto field,
redirect `OP_VARARG` to the table when present, drop the snapshot copy.

## Divergence 2b — `global function f() ... end` declaration form MISSING (clear-cut)

`global function NAME body` declares NAME as a global function (same
"already defined" guard as Div.1). Only a simple NAME is allowed (no `a.b.c`
fieldsel; upstream `globalfunc` calls `str_checkname` once,
`lparser.c:1947`).

| Code | ref 5.5.0 | lua-rs 5.5 |
|---|---|---|
| `global print; global function f() return 7 end; print(f())` | `7` | `variable 'global' not declared` (DIFF — `global` treated as a name) |
| `global print; global function f() end; global function f() end` | `global 'f' already defined` | (rejected earlier) |

- **Impl location**: the statement dispatcher at
  `crates/lua-parse/src/lib.rs:4373-4395`. The `is_global_decl` lookahead
  (line 4388-4389) accepts only `TK_NAME`, `'*'`, `'<'` after `global`; it must
  also accept `TK_FUNCTION`, and `globalstat` must branch to a `global function`
  path mirroring upstream `globalstatfunc`/`globalfunc`
  (`lparser.c:1947-1969`): declare the name as a regular global, compile the
  body, run the same already-defined check, store.
- **Clear-cut?** Yes; small parser change plus reuse of Div.1's guard.
  **CLEAR-CUT.**

## Divergence 3 — `<close>` on a global: wrong message text (clear-cut, trivial)

```
global print; global x <close> = setmetatable({},{})
  ref 5.5.0 : (command line):1: global variables cannot be to-be-closed
  lua-rs    : (command line):1: <close> is not allowed for a global declaration near '='
```

Both exit 1 (behavior correct); only the text differs. lua-rs (a) uses the
wrong wording and (b) appends a spurious `near '='`.

- **Expected**: `(command line):1: global variables cannot be to-be-closed`
  (NO `near` suffix — upstream `getglobalattribute`/`luaK_semerror` zeroes the
  token, `lparser.c:1862-1873`, `lcode.c:43-50`).
- **Impl location**: `crates/lua-parse/src/lib.rs:4119-4124` (the global `<close>`
  arm). Change the wording to `global variables cannot be to-be-closed` and emit
  it via a helper that adds the location but NOT the `near <token>` suffix
  (see Div.4 for the helper distinction). **CLEAR-CUT, TRIVIAL.**

## Divergence 4 — `unknown attribute` location/`near` formatting (clear-cut, trivial)

Two related text bugs from two different helpers:

```
local x <foo> = 1
  ref 5.5.0 : (command line):1: unknown attribute 'foo'
  lua-rs    : unknown attribute 'foo'              <-- MISSING location prefix
  (also DIFFs identically on 5.4)

global print; global x <foo> = 1
  ref 5.5.0 : (command line):1: unknown attribute 'foo'
  lua-rs    : (command line):1: unknown attribute 'foo' near '='   <-- spurious 'near'
```

Root cause: lua-rs has two syntax-error helpers with opposite formatting bugs:
- `LuaError::syntax(...)` (`crates/lua-types/src/error.rs:32`): adds NEITHER
  location nor `near`. Used by `getlocalattribute`
  (`crates/lua-parse/src/lib.rs:4022-4025`) — hence missing location on locals.
- `lua_lex::syntax_error(ls, msg)` (`crates/lua-lex/src/lib.rs:555`): adds
  location AND `near <token>`. Used by the global attribute arm
  (`lib.rs:4126`) — hence spurious `near`.

Upstream wants location + NO `near` for all of these (`luaK_semerror` sets
`ls->t.token = 0` to suppress the `near` suffix, `lcode.c:43-50`; `lexerror`
only appends `near` when the token is non-zero, `llex.c:118-123`).

- **Impl location**: `getlocalattribute` (`lib.rs:4009-4029`) and the global
  attribute arm (`lib.rs:4114-4131`). Both should emit `unknown attribute
  '<name>'` via a helper that adds `(command line):<line>:` but suppresses
  `near`. Consider a `syntax_error_nonear` (or `semerror`-style) helper analogous
  to upstream `luaK_semerror`. **CLEAR-CUT, TRIVIAL** — pure message text; verify
  with `diff_one.sh` on both 5.4 (`local x <foo> = 1`) and 5.5 (both local and
  global forms) so the local fix does NOT regress 5.4.

## Top 5 clear-cut fixes (priority order)

1. **Div.4 — `unknown attribute` message location/`near`** (trivial, also fixes
   5.4): give locals the location prefix, strip `near` from the global form.
2. **Div.3 — `<close>` global message text** (trivial): reword to
   `global variables cannot be to-be-closed`, no `near`.
3. **Div.1 — `global '<name>' already defined` runtime guard** (clear-cut
   behavior; medium impl): live-nil check before storing a `global name = expr`
   initializer.
4. **Div.2b — `global function f()` form** (clear-cut; small parser change +
   reuse of Div.1 guard).
5. **Div.2a — named varargs `function f(...t)`** (clear-cut spec; largest impl —
   parser + vararg-table lowering). Pragmatic lowering: at entry, bind `t` to a
   `table.pack(...)`-equivalent table; `.n` field counts all args incl. nils.
