# 5.5 stdlib + error/traceback divergences (DISCOVER, read-only)

Probed `/tmp/lua-refs/bin/lua5.5.0` (cross-checked `lua5.4.7`, `lua5.3.6`) against
`LUA_RS_VERSION=5.5 target/debug/lua-rs`. Every row below is reproduced with
`specs/oracle/diff_one.sh` or direct invocation. The differential oracle's path/addr
normalization was respected.

Summary: **5 confirmed 5.5 divergences** in this lane (utf8.offset arity, collectgarbage
option set, the `<no error object>` conversion, the namewhat traceback rendering, and the
collectgarbage param API). `_VERSION`, the math/string/table/os/io/coroutine/utf8 rosters,
`table.create`, and the `(error object is a X value)` / `__tostring`-on-error rendering all
already MATCH and need no work.

---

## 1. utf8.offset — 5.5 returns 2 values (start AND end byte position). CLEAR-CUT.

5.5 `utf8.offset(s, n [, i])` returns the byte position where the char starts **and** the
byte position where it ends (inclusive). 5.4/5.3 return only the start.

```
lua5.5.0 -e 'local a,b=utf8.offset("héllo",3); print(a,b)'   -->  4  4
lua5.5.0 -e 'local a,b=utf8.offset("aébc",2);  print(a,b)'   -->  2  3   (é spans bytes 2-3)
lua5.5.0 -e 'print(select("#", utf8.offset("héllo",3)))'     -->  2
lua-rs   (same)                                              -->  4  nil   (arity 1)
```

Second-return rules, from `lua-5.5.0/src/lutf8lib.c` `byteoffset` (lines 215-224):
- push `posi+1` (start, 1-based) as before;
- if `s[posi]` is a multi-byte lead, advance `posi` over trailing continuation bytes,
  then push `posi+1` (the last byte of that char) as the 2nd value;
- one-byte char: 2nd value == 1st value;
- **failure path (char not found / out of range): push only `nil`, arity stays 1** — do NOT
  push a 2nd value (`lua-5.5.0/src/lutf8lib.c` lines 211-213).

Impl location: `crates/lua-stdlib/src/utf8_lib.rs` `byte_offset` (line 333). Today it always
`Ok(1)`. For 5.5: on the success branch, after pushing the start, compute the end position
(skip continuation bytes via the existing `is_cont_at`) and push it, returning `Ok(2)`.
The nil/not-found branch stays `Ok(1)`. Gate on `state.global().lua_version == V5_5`.

CI assertion: in-process wrapper (`multiversion_oracle.rs`), a `v55_*` test asserting
`utf8.offset("aébc",2)` yields `(2,3)`, `utf8.offset("héllo",3)` yields `(4,4)`, arity 2;
and the not-found case yields arity 1. Add a 5.4 guard asserting arity 1 (no regression).

Error messages (`#3 position out of bounds`, `initial position is a continuation byte`)
already match; only the missing 2nd return differs.

## 2. collectgarbage option set differs: 5.5 drops setpause/setstepmul, adds "param". CLEAR-CUT.

5.4 (and lua-rs today) accept `setpause`/`setstepmul`. 5.5 **removed** them and added
`"param"`.

```
lua5.5.0 -e "print(pcall(collectgarbage,'setpause',100))"
  -->  false  bad argument #1 to 'collectgarbage' (invalid option 'setpause')
lua-rs 5.5  -->  true  50        (still accepted; WRONG for 5.5)
lua5.4.7    -->  true  200       (still valid in 5.4 — must NOT change)
```

The OPTS list in `crates/lua-stdlib/src/base.rs` `collectgarbage_fn` (lines 390-399) is
shared across all versions and hard-codes the 5.4 set. It must be version-gated:
- 5.3/5.4 keep `setpause`,`setstepmul` (and `incremental`/`generational` take numeric args).
- 5.5 valid options (`lbaselib.c` lines 201-204):
  `stop, restart, collect, count, step, isrunning, generational, incremental, param`
  — no `setpause`/`setstepmul`.

CI assertion: in-process wrapper. 5.5 test: `pcall(collectgarbage,'setpause',100)` is
`false` with `invalid option`. 5.4 guard: same call is `true` (no regression).

## 3. collectgarbage("param", name [, value]) — new 5.5 GC param API. CLEAR-CUT (shape), RISKY (values).

```
lua5.5.0 -e "print(collectgarbage('param','pause'))"        -->  250   (current value)
lua5.5.0 -e "local o=collectgarbage('param','pause',200); print(o)"  -->  250 (returns OLD, sets new)
lua5.5.0 -e "print(pcall(collectgarbage,'param','bogus'))"
  -->  false  bad argument #2 to 'collectgarbage' (invalid option 'bogus')
lua-rs 5.5  -->  bad argument #1 (invalid option)   (param not implemented at all)
```

From `lbaselib.c` lines 236-247: `param` reads arg2 via `luaL_checkoption` against
`{"minormul","majorminor","minormajor","pause","stepmul","stepsize"}`, reads optional arg3
(`luaL_optinteger`, default -1 = "just read"), and pushes the OLD value as an integer.

Observed current values on the reference (`lua5.5.0`):
`minormul=20, majorminor=50, minormajor=68, pause=250, stepmul=200, stepsize=9600`.

CLEAR-CUT: add the `"param"` option, the 6 param names, the invalid-name message, the
read-returns-current / write-returns-old contract, arity 1.
RISKY: the actual numeric values depend on lua-rs's GC internals. The existing GC param
plumbing in `base.rs` (`gc_set_param`, `gc_inc`, `gc_gen`) is flagged `TODO(port): stub in
Phase A` and already returns wrong values on 5.4 (`setpause` returns 50, ref 200 — see Note
B). Wiring exact 5.5 param values requires real GC param storage in `crates/lua-gc`. If the
GC backend can't store these, a faithful-shape stub (store the 6 ints in the state, read/
write them, default to the upstream-observed values above) satisfies the option/arity/
message contract without claiming a real GC tuning effect.

Impl: `base.rs` `collectgarbage_fn` (add `GcOp::Param` arm + the param-name `check_arg_option`)
plus param storage in `crates/lua-gc`. Gate the arm on 5.5.

CI assertion: in-process wrapper. 5.5: `collectgarbage('param','pause')` is an integer;
`pcall(collectgarbage,'param','bogus')` is `false`+`invalid option`. Asserting exact numeric
values is brittle — assert type/arity and round-trip (write X then read returns X) instead.

## 4. error(nil)/error() under a protected call -> "<no error object>" string (5.5-only). CLEAR-CUT.

5.5 converts a **nil** error object to the string `"<no error object>"` whenever an error
propagates (any pcall/xpcall, and the CLI top level). 5.3/5.4 leave it `nil`.

```
lua5.5.0 -e 'local ok,e=pcall(function() error(nil) end); print(ok,type(e),tostring(e))'
  -->  false  string  <no error object>
lua5.4.7 (same)  -->  false  nil  nil
lua-rs 5.5       -->  false  nil  nil      (matches 5.4, WRONG for 5.5)
```

Source: `lua-5.5.0/src/ldebug.c` `luaG_errormsg` (lines 838-852) — after running the message
handler (if any), `if (ttisnil(top)) top = "<no error object>"`. This code block does **not
exist in 5.4's ldebug.c** (confirmed by grep), so it is purely a 5.5 addition.
Note the bare top-level CLI message `(error object is a nil value)` is a SEPARATE path
(the CLI msghandler in `lua.c`) and **already matches** in lua-rs across all versions — do
not touch that.

lua-rs has no `luaG_errormsg` chokepoint (errors ride `Result::Err`). The closest catch
sites are `crates/lua-vm/src/do_.rs` around line 958 (`set_error_obj` / the post-handler
recovery at lines 968-988) and the inline pcall handler in `crates/lua-vm/src/api.rs`
(`pcall_k`, ~line 1709). For 5.5: at the point the final error object is settled (after the
handler runs, before it becomes the pcall result / CLI message), if it is `Nil`, replace it
with the interned string `"<no error object>"`. Gate on 5.5. Must cover BOTH the bare-pcall
return and the xpcall-handler-returns-nil case.

CI assertion: needs BOTH harnesses.
- in-process wrapper: `pcall(function() error(nil) end)` 2nd return is the string
  `"<no error object>"` on 5.5, `nil` on 5.4 (regression guard). Same for `error()` no-arg.
- spawn-the-binary (`crates/lua-cli/tests/traceback_oracle.rs`): top-level `error(nil)` CLI
  message is `(error object is a nil value)` (already correct; pin it so the fix above
  doesn't break the CLI path).

## 5. namewhat traceback rendering: 5.5 uses code-name first ("[C]: in global 'error'"). 5.5-only. CLEAR-CUT.

5.5 reorders `pushfuncname`: it now prefers the **namewhat** (`global`/`field`/`method`/
`local`/`upvalue`) over the global-name lookup. 5.3/5.4 try the global-name lookup first, so
any name reachable as a global renders `function 'name'`.

```
lua5.5.0 -e 'error("boom")'  traceback line:  [C]: in global 'error'
lua5.4.7 (same)                               [C]: in function 'error'
lua-rs 5.5 (same)                             [C]: in function 'error'   (matches 5.4, WRONG for 5.5)
lua5.4 MATCHes lua-rs 5.4 — do NOT change the 5.4 rendering.
```

This affects ALL frames, not just `error`. Confirmed full table on 5.5:
| call form                  | 5.3/5.4 render        | 5.5 render            |
|----------------------------|-----------------------|-----------------------|
| global C fn `error('x')`   | `in function 'error'` | `in global 'error'`   |
| global Lua fn `G()`        | `in function 'G'`     | `in global 'G'`       |
| table field `t.f()`        | `in field 'f'`        | `in field 'f'`        |
| method `t:f()`             | `in method 'f'`       | `in method 'f'`       |
| local `e()`                | `in local 'e'`        | `in local 'e'`        |
| upvalue                    | `in upvalue 'boom'`   | `in upvalue 'boom'`   |
(5.3/5.4 only diverge for things reachable as a global; non-global namewhats already match
because the global lookup fails and they fall through to namewhat.)

Source: `lua-5.5.0/src/lauxlib.c` `pushfuncname` (lines 96-110) vs 5.4's. 5.5 order:
`namewhat -> main -> pushglobalfuncname -> <file:line> -> '?'`. 5.4 order:
`pushglobalfuncname -> namewhat -> main -> <file:line> -> '?'`.

Impl location: `crates/lua-stdlib/src/auxlib.rs` `push_func_name` (lines 201-227). It
currently hard-codes the 5.4 order (global lookup first, lines 206-213). For 5.5, branch on
`lua_version == V5_5`: try `namewhat` first (the `else if !ar.namewhat.is_empty()` block at
line 214), then fall back to the global-name lookup. Keep the existing 5.4 order on the
5.3/5.4 path. The `method`-colon special-casing in the traceback builder (line 345) and
`push_global_func_name` itself are unchanged — only the ordering in `push_func_name` flips.

CI assertion: spawn-the-binary harness (`crates/lua-cli/tests/traceback_oracle.rs`) — it's a
traceback/namewhat behavior.
- 5.5: `error('boom')` traceback contains `[C]: in global 'error'`; a global Lua fn renders
  `in global '<name>'`.
- 5.4 guard: `error('boom')` traceback contains `[C]: in function 'error'` (no regression).

## Things that ALREADY MATCH (no work)

- `_VERSION` == "Lua 5.5" (and 5.4 == "Lua 5.4"). Correct.
- Library rosters identical 5.4<->5.5 for string/os/io/coroutine/utf8 and base `_G`.
- `math` roster: 5.5 default build drops `atan2,cosh,log10,pow,sinh,tanh` (LUA_COMPAT_MATHLIB
  off); lua-rs 5.5 already drops exactly those and 5.4 keeps them. Matches both binaries.
- `table.create` present on 5.5, absent on 5.4. lua-rs already matches.
- `(error object is a <type> value)` top-level CLI message for nil/table/boolean, and the
  `__tostring`-metamethod-on-error-object rendering — all match across versions.
- utf8.offset error messages (position out of bounds / continuation byte) match.
