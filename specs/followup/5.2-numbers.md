# Lua 5.2 numbers — float-only design (oracle = lua5.2.4)

Area: the float-only number model for `LuaVersion::V52`. READ-ONLY design pass.
Oracle = `/tmp/lua-refs/bin/lua5.2.4` (cross-checked vs lua5.4.7). All rows below
were reproduced live; the lua-rs column is from a scratch `is_supported(V52)=true`
build (reverted).

## Core insight — Int vs Float is mostly INVISIBLE once tostring is fixed

5.2 has one `number` type (`LUA_NUMBER=double`). lua-rs keeps the dual
`LuaValue` enum but must enforce float-only *behavior*. The crucial empirical
finding: for an **integer-valued** number, `Int(7)` and `Float(7.0)` are
**observationally identical** in 5.2 once `tostring` drops the `.0` — concat,
`%d` truncation, table keys, string indices (`("hi"):sub(2.0,4)`), `#t`, and
`for` counters all behave the same whether the value is carried as Int or Float.

So the gate-based (no-LuaValue-fork) approach is **sufficient** for 5.2. There
are only **four** places where an Int leak is actually observable, and a fifth
non-number roster item:

1. `tostring`/`print`/concat of an integer-valued float prints `5.0` not `5`.
2. **Negative zero**: `0*-1` must be `-0` (i.e. `-0.0`), not `0`.
3. `string.format("%d", x)` with a non-integral float: 5.2 **truncates and
   succeeds**; 5.3+ (current code) **errors**.
4. `math.type`/`tointeger`/`maxinteger`/`mininteger` must be absent.
5. (adjacent, not numbers) the 5.2 math roster add-backs + `bit32`.

## The seams (production sites) and the fix at each

### (a) Lexer — never emit Int  [`crates/lua-lex/src/lib.rs` `read_numeral` ~L1034]
`read_numeral` calls `lua_vm::object::str2num` then maps `Int→TK_INT`,
`Float→TK_FLT`. It already has `state: &mut LuaState`, so it can read
`state.global().lua_version` (the file already does this for the `is_v53`
codepoint check at ~L1323). **Fix:** when `number_model()==FloatOnly`, coerce a
returned `Int(i)` to `Float(i as f64)` and emit `TK_FLT`. Hex integer literals
(`0x10`) and big literals (`9007199254740993`) then become floats, matching
5.2.4 (`0xffffffffffffffff` → `1.844674407371e+19`).

This single fix has a **large downstream payoff**: the parser's compile-time
constant folder (below) keys off `ExprKind::KInt`/`KFlt`, which come straight
from the lexer token. If the lexer never produces `KInt`, the folder's
`(KInt,KInt)→Int` branch is **naturally dead** under V52 and folding flows
through the float `promote` path → `KFlt`. **No parser version-threading
needed.** This is why the gate approach stays small.

### (b) Arithmetic — never PRODUCE an Int  [`crates/lua-vm/src/object.rs` `raw_arith` L130-185]
`raw_arith` is the one central dispatch (used by VM ops, `api::arith`,
`state::arith_op`, and metamethod paths). Its `_ =>` arm does
`(Int,Int)→Int_arith→Int`. With seam (a), runtime Int operands are rare, but
they still arise: `string.len`, `#t`, `select('#',...)`, `os.time()`, table
counts, `string.byte`, loop internals — all hand back `Int`. `7 % 3` on two
such Ints would yield `Int`. Observationally that's invisible **except** for
negative zero and for any value that later hits a float-only formatting path.
The neg-zero failure (`0*-1`) is the proof a leak is real.
**Fix:** in `raw_arith`, when `state.global().lua_version.number_model() ==
FloatOnly`, skip the `(Int,Int)→Int` fast path and the bitwise/Bnot/Shl/Shr
integer ops are unreachable anyway (5.2 has no bitwise operators — those tokens
don't lex), routing everything through `float_arith`. `state` is already in
scope. (`Div`/`Pow` already always float.) Cheapest correct form: gate just the
`_ =>` arm's int-int branch.

NOTE on `//` (IDiv): 5.2 has no `//` operator, so the token never lexes; no
runtime gate needed for the operator. The compile-time folder's `IDiv` branch
is likewise unreachable under V52 (no KInt operands).

### (c) tostring/concat — suppress `.0` on integer-valued floats  [`crates/lua-vm/src/object.rs` `number_to_str_buf` L591-622]
This is the dominant fix and is **exactly what spike commit d8ec33f did** (that
commit is NOT on this branch — current `number_to_str_buf` still appends `.0`
unconditionally). Port the spike's change: thread `LuaVersion` (not a bool) into
`number_to_str_buf`; gate the `looks_like_int → push ".0"` with
`dual_model = !FloatOnly`. The two callers (`num_to_string` L637, `addnum2buff`
L733) already fetch `state.global().lua_version`. Float formatting stays `%.14g`
(`fmt_g(*f,14)`) for 5.1–5.4 — verified: `1/3`→`0.33333333333333`,
`1e16`→`1e+16`, `12345.678` exact. Negative zero prints `-0` (Rust `{:e}`/`fmt_g`
preserves the sign bit — confirm in the impl; lua5.2.4 prints `-0`).

### (d) math roster  [`crates/lua-stdlib/src/math_lib.rs`]
- **Remove under V52**: `type`, `tointeger`, `maxinteger`, `mininteger`. The
  spike already gated `math.type`/`tointeger`/`maxinteger`/`mininteger` out for
  V51/V52 (commit d8ec33f, math_lib.rs +13). Reuse that gate. (`math_type` at
  L548 has an `is_v53` check already; `maxinteger` reg is `None` at L748 with a
  version delta block at L794+.)
- `math.floor`/`ceil`/`modf` return a number (float) — already fine, no int
  subtype to honor.

### (e) string.format `%d` / `%i` strictness  [stdlib `str_lib`/`format`]
5.2 truncates a non-integral float toward zero for `%d`/`%i`/`%u`/`%x`/`%o`
(`%d` of `3.5`→`3`, `-3.5`→`-3`, `2.9`→`2`); 5.3+ raises "number has no integer
representation". The current code raises (5.3+ behavior). **Fix:** under
FloatOnly, convert the `%d`-family argument with truncation (C `(lua_Integer)d`)
instead of the exact-integer check. Find the `format` integer-arg path and gate
on `number_model()`.

## Adjacent (NOT strictly numbers, but blocks a faithful 5.2) — flag

- **math roster ADD-BACKS**: 5.2 has `math.log10`, `atan2`, `pow`, `cosh`,
  `sinh`, `tanh` (and `ldexp`/`frexp`, which lua-rs already has). lua-rs V52
  currently lacks `log10/atan2/pow/cosh/sinh/tanh`. `math.log(x,base)` 2-arg
  form works in 5.2.4 (`math.log(8,2)`→`3`) — confirm lua-rs honors it under V52.
- **`bit32`** library is present in 5.2.4 (`bit32.band/bor/...`), absent in
  lua-rs V52. Needed for roster parity.
- **`fenv` absent** (`getfenv`/`setfenv` are nil in 5.2.4) — lua-rs already lacks
  them, so 5.2 is fine here; this is the whole reason 5.2 (not 5.1) is the bridge.
- `table.pack`/`table.unpack`/`unpack` present; `utf8` absent (5.3+). Verify.

## Is the gate approach sufficient for numbers? YES.

No case in the numbers area forces a deeper change or a `LuaValue` fork. The
dual enum is retained; four small version gates at the production sites (a,b,c,e)
plus the roster removal (d) reproduce every probed lua5.2.4 number behavior. The
neg-zero case is the only subtle one and is handled for free once arithmetic is
float-only (seam b) — no special casing.

## Top fixes (priority order)

1. **(c) tostring `.0` suppression** — port spike d8ec33f's `number_to_str_buf`
   version-threading. Highest-impact, fixes the majority of the diff table.
2. **(a) lexer float-only** — coerce Int→Float in `read_numeral` under FloatOnly;
   makes parser const-folding float-only for free (no parser changes).
3. **(b) arith float-only** — gate `raw_arith`'s `(Int,Int)→Int` arm; fixes
   negative zero and stops residual Int production.
4. **(e) `%d` truncation** — gate `string.format` integer-arg path to truncate
   (5.2) instead of erroring (5.3+).
5. **(d) math roster removal** — gate out `type/tointeger/maxinteger/mininteger`
   (spike already shows how). Then the roster ADD-BACKS + `bit32` (adjacent area).
